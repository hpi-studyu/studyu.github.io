# JSON Schema Reference — Overview

This section documents the complete JSON structure of every model in the StudyU system. Use this as a reference when reading database records, building API integrations, or debugging serialization issues.

## Serialization conventions

| Convention | Detail |
|---|---|
| Key format | `snake_case` (matching PostgreSQL column names) |
| Field names in Dart | `camelCase` — mapped via `@JsonKey(name: '...')` annotations |
| Nullable omission | `null` fields may be omitted entirely (`includeIfNull: false` in some classes) |
| Default values | Applied during deserialization if a key is missing — defined via `@JsonKey(defaultValue: ...)` |
| DateTime format | ISO-8601 strings (e.g., `"2024-03-15T08:30:00.000Z"`), always stored in UTC |
| Time-of-day format | `"HH:MM"` 24-hour (e.g., `"08:00"`, `"20:30"`) — custom `StudyUTimeOfDay` serialization |
| Color format | ARGB integer (e.g., `4278190335` = `0xFF0000FF` = blue) |
| UUID format | Standard UUID v4 strings (e.g., `"a1b2c3d4-e5f6-7890-abcd-ef1234567890"`) |

## Polymorphic deserialization

Classes with a `type` discriminator field use factory constructors with `switch` expressions. The `type` string must match exactly (case-sensitive).

```dart
// Example from Question.fromJson
switch (json['type']) {
  case 'boolean': return BooleanQuestion.fromJson(json);
  case 'choice': return ChoiceQuestion.fromJson(json);
  // ...
}
```

## Generated files

All models have corresponding `.g.dart` files generated by `build_runner` with `json_serializable`. If you see a serialization error, regenerate with:

```bash
melos run generate
```
