# Coding Conventions

## Dart style

- Follow [Effective Dart](https://dart.dev/guides/language/effective-dart) style guidelines
- Run `melos format` before every commit — CI will reject unformatted code
- Run `dart analyze` (or `melos run qualitycheck`) — CI will reject code with analysis warnings
- Generated files (`*.g.dart`) must be committed with the model changes that produced them
- Do not add `dart-define` values or secrets to committed `.env` files
- Prefer the `dev` backend over production for all development and testing

## Commit message format

This project uses [Conventional Commits](https://www.conventionalcommits.org):

```
feat(designer): add eligibility criteria builder
fix(app): correct phase duration calculation
chore: upgrade melos to 7.3.0
docs: update local setup instructions
```

**Types:** `feat`, `fix`, `chore`, `docs`, `test`, `refactor`, `ci`

**Scope** (optional) should be the package or feature area: `app`, `designer`, `core`, `flutter_common`, `database`.

## Branch strategy

- Base feature branches on `dev`, not `main`
- Open pull requests against `dev`
- `main` receives merges from `dev` on release

## Generated files policy

All `*.g.dart` files generated by `build_runner` must be committed together with the model changes that produced them. This is because `core` is a path dependency — the other packages cannot import it without the generated files being present on disk.

Do not commit generated files as a separate commit from the model changes. Stage them together:

```bash
# After melos run generate
git add core/lib/src/models/  # model changes
git add core/lib/src/models/  # *.g.dart files (same directory)
git commit -m "feat(core): add new field to Study model"
```
